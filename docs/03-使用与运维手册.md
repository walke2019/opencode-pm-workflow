# pm-workflow 使用与运维手册

## 1. 安装与插件接入

### 安装

```bash
npm install @walke/opencode-pm-workflow
```

### 离线 CLI（pmw，0.8.0 起）

全局安装后可用 `pmw` 命令在任何项目跑诊断与预演，无需 OpenCode 在线：

```bash
npm install -g @walke/opencode-pm-workflow

pmw --help
pmw doctor                                       # 当前目录的 doctor 报告
pmw doctor --cwd /path/to/project --json         # 指定目录 + JSON 输出
pmw dispatch dry-run '修复登录 401'              # dispatch 预演（不执行）
pmw state --json                                 # state.json 摘要
pmw history --limit 5 --type fallback.foreground_switch
pmw report                                       # 生成 .pm-workflow/report.html
pmw report --out ./dispatch-report.html          # 自定义路径
pmw agents list                                  # 列项目级 + 全局级 agent
pmw agents promote pm_lead --overwrite           # 把项目级 agent 复制到全局（覆盖）
pmw agents doctor --json                         # 检查 frontmatter 完整性
pmw docs check                                   # 检查 README / 主文档 / Change Log 治理规则
pmw verify                                       # 本地跑 typecheck+build+smoke
```

CLI 默认只读，只有 `pmw verify` 会调本包 `npm run verify-release`，`pmw report` 会写一份 HTML 到本地，`pmw agents promote` 会写文件到 `~/.config/opencode/agents/`（XDG_CONFIG_HOME 隔离友好）。CI 场景推荐 `pmw doctor --json` 配合 `jq` 消费 blockers 字段做门禁，也推荐 `pmw docs check --json` 把 README 版本、主文档数量、Change Log 与旧路径引用纳入发布前检查；初次模型配置推荐填模板后交给 AI 读取合并；运维场景推荐 `pmw report` 生成本地报告排查 dispatch / fallback / auto-continue 链路（不联网，不上传，浏览器直接打开）；多项目复用 agent 推荐 `pmw agents list / promote / doctor` 三件套维护跨项目 agent 库。

### OpenCode 接入

推荐使用 OpenCode 官方 npm plugin 配置方式，在项目级或全局 `opencode.json` 中声明：

```json
{
  "$schema": "https://opencode.ai/config.json",
  "plugin": ["@walke/opencode-pm-workflow"]
}
```

OpenCode 会在启动时自动安装并缓存 npm plugin。仅在需要本地调试包入口时，再在 OpenCode 项目的插件入口中引用 server / TUI 子路径：

```ts
// plugins/pm-workflow-plugin.ts
export { default } from "@walke/opencode-pm-workflow/server";
```

```ts
// plugins/pm-workflow-plugin-tui.ts
export { default } from "@walke/opencode-pm-workflow/tui";
```

> 注意：不要同时加载源码入口、dist 入口和兼容壳，避免插件重复注册。


## 4. 推荐操作顺序

```mermaid
flowchart LR
    A[查看状态] --> B[pm-check-project-state]
    B --> C[检查 Gate]
    C --> D[pm-check-gates]
    D --> E[预演调度]
    E --> F[pm-dry-run-dispatch]
    F --> G[执行调度]
    G --> H[pm-execute-dispatch confirm=YES]
    H --> I[诊断与回执]
    I --> J[pm-doctor / pm-get-last-execution]
```

### 日常使用推荐流程

```bash
# 1. 查看当前项目状态
pm-check-project-state

# 2. 检查 gate 是否通过
pm-check-gates

# 3. 预演调度（不实际执行）
pm-dry-run-dispatch

# 4. 确认无误后执行
pm-execute-dispatch confirm=YES
```

## 5. 诊断与排障

### 常见问题

#### Q: 系统提示 "is a subagent, not a primary agent. Falling back to default agent"

**原因**：specialist agent 被错误地按 `opencode run --agent` primary 路径调用。

**解决**：确保 subagent 通过 `opencode task ...` 执行，而不是 primary path。

#### Q: `npm test` 报 `ERR_MODULE_NOT_FOUND: Cannot find module 'dist/index.js'`

**原因**：并行执行 `npm test` 与 `npm run build`，`build` 的 `clean` 清掉了 `dist/`。

**解决**：改为串行执行：

```bash
npm run build && npm test
```

#### Q: 找不到 `node` / `npm` 命令

**原因**：当前 shell 环境 PATH 不含 Homebrew 路径。

**解决**：显式前缀：

```bash
PATH="/opt/homebrew/bin:$PATH" npm run build
PATH="/opt/homebrew/bin:$PATH" npm test
```

#### Q: 文档路径失效

**原因**：旧文档已被删除，但某些地方仍引用旧路径。

**解决**：现行文档仅 5 篇：
- `README.md`
- `docs/01-技术架构.md`
- `docs/02-业务功能与任务流转.md`
- `docs/03-使用与运维手册.md`
- `docs/04-待办与演进清单.md`

### 健康检查

```bash
# 全面健康检查
pm-doctor

# 自动修复（安全操作）
pm-doctor-repair

# 安全审计
pm-safety-report
```

## 8. FAQ

### Q: 我应该用哪个 Lane？

**A**：日常开发默认用 `pm-medium`。快速预览用 `pm-quick`，完整执行用 `pm-full`，定位问题用 `pm-debug`。

### Q: 新增一个 agent 文件后，需要改 pm-workflow 核心逻辑吗？

**A**：默认不需要。新 agent 先进入 `agents/*.md` 被 Registry 识别，只有当它代表一种**高频、稳定、边界清晰、值得自动分派**的新任务域时，才考虑进入核心语义层。

### Q: 自动续跑会绕过安全检查吗？

**A**：不会。自动续跑必须经过 Gate / Permission / Confirm 检查，只要任一条件不满足就会停住。

### Q: 模型配置应该怎么写？

**A**：初次使用推荐用模板，不要求用户记命令参数。把 `pm-workflow.models.example.json` 复制为你自己的模型配置；模板里 `agent_profiles` 段对每个 agent 都给出了角色画像，便于你按"该 agent 干什么"选合适模型：

| Agent | 模式 | 角色 | 主模型应具备 | 回退模型应具备 |
| --- | --- | --- | --- | --- |
| `pm_lead` | primary | 主协调官：分析决策、规划分派、收敛验收 | 强推理、长上下文、决策稳健、中文优先 | 低成本但保留中文与结构化输出 |
| `pm_advisor` | primary | 拆解顾问：把复杂任务拆成清晰步骤、识别风险 | 结构化拆解、风险识别、中文优先 | 低成本但结构化输出不丢 |
| `pm_backend` | subagent | 后端执行：API、数据库、服务、性能 | 编码能力强、调试推理、理解复杂依赖 | 仍能写出可运行代码 |
| `pm_frontend` | subagent | 前端执行：页面、组件、交互、响应式 | UI 直觉、CSS/样式准确 | 保留响应式与可访问性 |
| `pm_reviewer` | subagent | 审查与文档：测试、回归、code review、发布说明 | 细致审查、找 bug、文档语感 | 仍能完成检查表与发布说明 |
| `pm_researcher` | subagent | 调研：资料检索、官方方案、事实核查 | 检索能力、概要提炼、中英双语 | 保留检索与提炼能力 |

填好其中你关心的字段后（缺省字段会用 `default_model` / `default_fallback_model` 兜底），保存到任意位置，然后直接对 AI 说："读取这个模型模板，校验 OpenCode 全局模型清单，并合并到 pm-workflow 配置"。

AI 会按 `agent_profiles` 中的 `model_traits` 校验你填的模型是否合理；不合理时会给出 1-3 个候选让你确认，**不会静默替换**。最终它写入：

- `agents.definitions.<agent>.model`
- `agents.definitions.<agent>.fallback_models`
- `fallback.chains.<agent>`

模型清单从全局 OpenCode 配置 `~/.config/opencode/opencode.json` 的 `provider.*.models` 读取。OpenCode 官方 agent `model` 字段使用 `provider/model-id` 格式；若你的 provider.models 已经使用带 `/` 的完整模型 key（例如 `cx/gpt-5.5`），pm-workflow 会直接接受该 key。不要臆造模型 ID。

### Q: 如何判断当前项目处于哪个阶段？

**A**：运行 `pm-check-project-state` 或 `pm-get-state`，系统会返回当前 stage 与下一步建议。

### Q: Todo 没完成怎么办？

**A**：每个 todo 必须完成，或标注 blocked 并说明原因。Todo 是过程终结标准，不是可选项。

### Q: 怎么配置运行时模型降级（ForegroundFallback）？

**A**：在 `.pm-workflow/config.json` 或全局配置 `~/.config/opencode/pm-workflow.config.json` 的 `fallback` 节增加 `chains` 字段：

```jsonc
{
  "fallback": {
    "max_attempts": 1,
    "chains": {
      // 按 semantic agent 配置（推荐）
      "pm_lead": ["kr/claude-sonnet-4.5", "cx/gpt-5.5"],
      "pm_backend": ["cx/gpt-5.5", "kr/claude-sonnet-4.5"],
      // 也可按具体 model id 配置（按 agent 找不到时再按 model 找）
      "cx/gpt-5.5": ["kr/claude-sonnet-4.5"]
    }
  }
}
```

触发条件：dispatch 子进程返回非 0，且 stderr/stdout 命中 `429` / `rate-limit` / `timed out` / `context length exceeded` / `model not found` 等模式时，自动切到链路下一个 model 重试。`max_attempts` 控制最多切换次数。降级事件写入 `.pm-workflow/history.jsonl` 的 `fallback.foreground_switch` 类型。

### Q: 量化分派指引（agentStats）什么时候出现在 handoff prompt？

**A**：仅当 `analysis.fallbackAgents.length >= 1`（任务有多候选 agent）时，handoff packet 自动注入 1-3 张候选卡片，包含速度/成本/质量/适用条件等量化字段。单候选场景不注入，避免无意义 token 消耗。这是 prompt 工程层面的优化，不需要任何配置。

### Q: 怎么打开 0.5.0 的 Auto-continue（自动续跑）？

**A**：默认是关闭的。需要在 `.pm-workflow/config.json` **同时**打开两个总开关：

```jsonc
{
  "permissions": {
    "allow_auto_continue": true
  },
  "auto_continue": {
    "enabled": true,
    "max_steps": 3,
    "cooldown_ms": 2000,
    "require_clean_tree": false,
    "stop_on_feedback_signal": true
  }
}
```

打开后仍受 Gate / Permission / Confirm 全部约束；任一不满足都会停。

### Q: Auto-continue 链路被中途打断怎么办？

**A**：所有终止情况都会写到 `.pm-workflow/history.jsonl`，可以用 `pm-get-state` 查看 `state.auto_continue.aborted_reason`，或翻 history 找：

- `auto_continue.chain_start`：链路启动
- `auto_continue.step`：单步成功
- `auto_continue.aborted`：被终止（含 `reason` 字段：`guard-blocked` / `gate-blocked` / `feedback-stop` / `execution-failed`）

### Q: 我说"停下"为什么 specialist 还在跑？

**A**：`stop_on_feedback_signal` 检测的是 **specialist 输出**里的停止词，而不是用户输入。如果你想从主对话里打断当前 specialist，请直接 `Esc Esc` 或关闭 OpenCode session。下一轮 dispatch 会重新评估 Auto-continue Guard。

### Q: 怎么看插件健康检查（0.6.0）输出？

**A**：插件装配时自动跑 `evaluatePluginHealth`，结果写到 OpenCode 的 `app.log`：

- 一切达标 → `info: [health] pm-workflow plugin assembly satisfies all thresholds`
- agents 不足 → `warn: [health] pm-workflow expected at least 1 agent(s) but found 0`
- tools 不足 → `warn: [health] pm-workflow expected at least 5 tool(s) but found N`
- mcps 不足 → `info: [health] ...`（不强制约束）

**自定义阈值**：在 OpenCode `plugin` 装载选项里传 `health: { minAgents: 6, minTools: 20 }`。

### Q: hot-reload 后日志显示 `activation: duplicate` 是什么意思？

**A**：0.6.0 起，pm-workflow 插件在同一进程内同一 plugin id 只允许装配一次，避免事件回调被重复注册导致 `syncState` / `writeReviewMarker` 执行多遍。重复装配时仍提供完整的 tool / config 集合（无副作用），但跳过 hooks 注册。这是预期行为，不影响功能。

### Q: 怎么用 frontmatter 配置 `pm_lead` 能调哪些 subagent？（0.7.0）

**A**：在 `.opencode/agents/pm_lead.md` 顶部写：

```yaml
---
description: PM 主协调官
mode: primary
permission:
  task:
    pm_backend: allow
    pm_frontend: allow
    pm_reviewer: ask
    pm_researcher: deny
---
```

**优先级**：

1. `permission.task[next]=deny` → 立即拒绝；写 `routing.denied` 历史
2. `permission.task[next]=allow|ask` → 立即允许
3. 没声明 / 没 frontmatter → 按 fallback（默认允许，向后兼容）

被拒绝的 next agent 不会被 Auto-continue 选中；链路停在 `completed` 状态。

### Q: frontmatter 写错了会怎样？

**A**：`agent-routing.ts` 的 parser 是容错的——解析失败的行直接跳过，不会拖垮 dispatch。但你的路由意图也不会生效。建议同时跑 `pm-doctor` 检查路由解析结果，或临时打开 OpenCode 日志看 `routing.denied` 事件。
