## 3. 已确认边界

### 核心任务域（当前集合）

- `commander`
- `plan`
- `frontend`
- `backend`
- `researcher`
- `writer`
- `qa_engineer`
- `tech-lead`（窄定义：仅技术评审/架构审查/风险判断）

### 禁止扩张规则

1. **新增 agent ≠ 新增语义角色**
2. **核心任务域保持少量稳定**
3. **优先优化已有边界，不优先加新角色**
4. **平台型/岗位型/万能型角色不直接进入核心任务域**
5. **模型差异属于执行策略，不属于 workflow 语义**
6. **复合任务通过编排解决，不通过新增角色解决**

### 架构治理原则

> **pm-workflow 识别的是"工作类型"，不是"你当前配置了多少个 agent"。**
>
> **agent 可以增长，但核心语义不应随之线性膨胀。**

## 6. 中期演进方向

### 可能考虑的角色（需满足证据门槛）

以下角色**当前不进入核心任务域**，只有同时满足以下条件才值得考虑：

1. 高频出现
2. 长期稳定
3. 边界一句话能说清
4. 与现有角色显著不同
5. 自动分派收益明显
6. 不会显著提升歧义成本

| 候选角色 | 当前状态 | 说明 |
| --- | --- | --- |
| `security` | 观察中 | 可能与 qa / tech-lead / backend 重叠 |
| `ops` | 观察中 | 当前主场景仍是开发编排，非运维 |
| `data` | 观察中 | 多数数据任务仍可落在 researcher / backend |

### 不建议进入核心的角色

- `android` / `ios` / `mobile` → 更像 frontend 子域
- `architect`（泛定义） → 与 tech-lead 重叠
- `principal-engineer` / `senior-frontend` → 组织岗位名，非工作类型
- `specialist` / `expert` / `consultant` → 万能桶角色，边界模糊

## 7. 长期路线（不承诺时间，只承诺方向）

以下方向**当前不在任何里程碑里**，每一项落地前都会先发一份 RFC 文档（写进本节"详细 RFC"或独立章节），等用户确认价值后才进 0.x.0 路线。

### 7.1 `@walke/opencode-pm-workflow-cli` 子包 ✅ 已落地（0.8.0，同包 bin 形态）

**实际落地形态**：未拆独立子包，改为同包内 `bin: { pmw: "./scripts/cli/index.mjs" }`，原因是子包发布维护成本高，`bin` 字段已能让用户直接通过 `pmw` 命令调用，效果一致而维护面更小。

**已交付命令**：

- `pmw doctor [--json]` — 输出当前项目 doctor 报告
- `pmw dispatch dry-run [prompt...]` — dispatch 预演，不执行命令
- `pmw state [--json]` — 输出 state.json 摘要
- `pmw history [--limit N] [--type T] [--json]` — 查询 history.jsonl
- `pmw verify` — 本地跑 typecheck + build + smoke + pack-dry-run
- 全局支持 `--cwd <path>` / `--json` / `--help` / `--version`

**实现要点**：复用 dist/ 中已纯函数化的 `buildDoctorReport` / `buildDispatchCommand` / `buildExecutionPlan` / `buildStateSummary` / `queryHistory`；零额外依赖、零运行时改动。CI / 服务器 / 没装 OpenCode 的环境直接可用。

### 7.2 可视化执行回执 dashboard ✅ 已落地（0.9.0，单文件 HTML）

**实际落地形态**：通过 `pmw report` 子命令生成单文件 HTML 静态报告，无 server、无端口、无外链字体、无前端框架；嵌入内联 CSS（深色主题）+ vanilla JS 筛选。

**已交付能力**：

- 关键指标卡片：dispatch 总数 / 失败数、fallback 切换、auto-continue 链 / 步 / 中止、routing 拒绝
- 事件类型分布表（按数量倒序）
- 事件流可筛选（按类型 / 文本搜索 / 最近 N 条）
- XSS 转义内置：事件 JSON 嵌入 script 标签时 `<` → `\u003c`，恶意 HTML 不会被解释
- 体积：~8 KB（空项目）/ 与事件数量线性相关
- 输出可定制：默认 `.pm-workflow/report.html`，`--out <path>` 自定义；`--json` 仅输出 summary 不写文件

**实现要点**：复用 `dist/core/history.js` 的 `readHistory`；纯本地计算与渲染；不上传任何数据；浏览器打开报告也不发任何外部请求。

### 7.3 跨项目共享 agent 库 ✅ 已落地（0.10.0，三件套 CLI）

**实际落地形态**：通过 `pmw agents list / promote / doctor` 三个子命令实现，复用 dist/ 中的 `src/core/agent-library.ts` 模块。

**已交付能力**：

- `pmw agents list [--json]` — 列出项目级 + 全局级 agent，识别"项目覆盖全局"shadow 关系；同名时项目优先，不重复列。每条记录显示 mode / model / 是否含 permission.task / 当前 finding 数。
- `pmw agents promote <id> [--overwrite] [--json]` — 把项目级 agent 复制到 `~/.config/opencode/agents/`；默认拒绝覆盖、不删原文件，需显式 `--overwrite` 才覆盖；XDG_CONFIG_HOME 隔离友好（多账户 / CI 测试场景）。
- `pmw agents doctor [--json]` — 检查所有 agent 的 frontmatter 完整性，4 类 finding（缺 description / mode / model / primary 缺 permission.task），分级（warn / info）+ 聚合计数 + 明细。findings 是建议而不是错误，永远 exit=0。

**实现要点**：自研轻量 frontmatter 字段抽取，不引入 yaml 依赖；与 0.7.0 `agent-routing.ts` 共享 `parseFrontmatterTaskPermission`；不修改 dispatch_map 与运行时路由。

### 7.4 永不进入路线的方向

以下能力**永不**作为 pm-workflow 路线项，无论用户呼声多高（详见 §5）：

- Council 多模型并行投票
- Multiplexer / 终端接管
- Divoom / 硬件状态显示器
- 纯 prompt-based 编排（放弃 Gate 安全约束）

这些方向不属于"等条件成熟",而是与 pm-workflow 的核心安全 / token 节省原则冲突。

## 8. 文档治理与架构治理规则

### 文档治理

1. **现行真相只保留一处**：同一机制的当前版本说明只能在主文档中存在一份权威描述
2. **文档按读者任务组织**：不按历史 spec/plan/migration 组织
3. **流程图服务正文**：不再散落维护，合并到主文档
4. **总量控制**：README + 4 篇主文档，不新增第 6 篇现行主文档

### 架构治理

1. **Analyzer 只负责语义判断**：不决定真实 agent 文件来源
2. **Registry 只负责执行绑定**：不发明新任务语义
3. **Runtime 只负责执行编排**：不做二次语义分类
4. **新增 agent 先进入 Registry，再观察是否值得进入 Analyzer**
5. **角色进入核心任务域必须经过证据门槛**

### PR 审查标准

任何涉及角色、路由、调度的变更，都必须回答：

1. 该变更是在修正已有任务域边界，还是在新增任务域？
2. 该新增对象是稳定工作类型，还是仅仅是 agent 名称/岗位名称/平台名称？
3. 该变更是否减少了多处真相？
4. 该变更是否让 Runtime 更纯粹？
5. 该变更是否会导致"每加一个 agent 就想加一套语义支持"？
