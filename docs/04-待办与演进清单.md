## 3. 已确认边界

### 核心任务域（当前集合）

- `commander`
- `plan`
- `frontend`
- `backend`
- `researcher`
- `writer`
- `qa_engineer`
- `tech-lead`（窄定义：仅技术评审/架构审查/风险判断）

### 禁止扩张规则

1. **新增 agent ≠ 新增语义角色**
2. **核心任务域保持少量稳定**
3. **优先优化已有边界，不优先加新角色**
4. **平台型/岗位型/万能型角色不直接进入核心任务域**
5. **模型差异属于执行策略，不属于 workflow 语义**
6. **复合任务通过编排解决，不通过新增角色解决**

### 架构治理原则

> **pm-workflow 识别的是"工作类型"，不是"你当前配置了多少个 agent"。**
>
> **agent 可以增长，但核心语义不应随之线性膨胀。**

## 5. 中期演进方向

### 可能考虑的角色（需满足证据门槛）

以下角色**当前不进入核心任务域**，只有同时满足以下条件才值得考虑：

1. 高频出现
2. 长期稳定
3. 边界一句话能说清
4. 与现有角色显著不同
5. 自动分派收益明显
6. 不会显著提升歧义成本

| 候选角色 | 当前状态 | 说明 |
| --- | --- | --- |
| `security` | 观察中 | 可能与 qa / tech-lead / backend 重叠 |
| `ops` | 观察中 | 当前主场景仍是开发编排，非运维 |
| `data` | 观察中 | 多数数据任务仍可落在 researcher / backend |

### 不建议进入核心的角色

- `android` / `ios` / `mobile` → 更像 frontend 子域
- `architect`（泛定义） → 与 tech-lead 重叠
- `principal-engineer` / `senior-frontend` → 组织岗位名，非工作类型
- `specialist` / `expert` / `consultant` → 万能桶角色，边界模糊

## 6. 文档治理与架构治理规则

### 文档治理

1. **现行真相只保留一处**：同一机制的当前版本说明只能在主文档中存在一份权威描述
2. **文档按读者任务组织**：不按历史 spec/plan/migration 组织
3. **流程图服务正文**：不再散落维护，合并到主文档
4. **总量控制**：README + 4 篇主文档，不新增第 6 篇现行主文档

### 架构治理

1. **Analyzer 只负责语义判断**：不决定真实 agent 文件来源
2. **Registry 只负责执行绑定**：不发明新任务语义
3. **Runtime 只负责执行编排**：不做二次语义分类
4. **新增 agent 先进入 Registry，再观察是否值得进入 Analyzer**
5. **角色进入核心任务域必须经过证据门槛**

### PR 审查标准

任何涉及角色、路由、调度的变更，都必须回答：

1. 该变更是在修正已有任务域边界，还是在新增任务域？
2. 该新增对象是稳定工作类型，还是仅仅是 agent 名称/岗位名称/平台名称？
3. 该变更是否减少了多处真相？
4. 该变更是否让 Runtime 更纯粹？
5. 该变更是否会导致"每加一个 agent 就想加一套语义支持"？
