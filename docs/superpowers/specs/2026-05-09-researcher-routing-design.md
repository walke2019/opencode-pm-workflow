# Researcher 语义路由设计

日期：2026-05-09  
状态：已确认，待实现计划  
范围：`src/orchestrator/analyzer.ts`、相关测试、少量运行说明文档

## 背景

当前 pm-workflow 已完成以下基础能力：

1. lane-aware orchestration 已落地。
2. mode-aware dispatch 已修复 subagent 被 primary 路径误调用的问题。
3. Agent Definition Registry 已接入 runtime dispatch，agent 的 `model / mode / description` 优先来源于真实 `opencode/agents/*.md`。

在此基础上，当前最明显的剩余缺口不在“如何执行 agent”，而在“如何把任务语义更准确地交给合适角色”。

用户已明确要求当前阶段优先补齐：

- `researcher` 的独立语义识别

并已明确确认该角色的触发策略采用：

- **B：中等触发**

即：不仅匹配显式“调研 / 搜索 / 查资料 / 官方文档 / 对比方案”等关键词，也匹配“帮我看看业内怎么做 / 查一下官方推荐 / 对比几种方案”这类以信息搜集、资料核验、外部事实比对为主目标的研究型请求；但不能过度扩张，抢占本应交给实现、规划、评审、文档、测试角色的任务。

## 目标

1. 为 analyzer 增加 `researcher` 域的独立语义识别。
2. 让明显的资料搜集、外部信息核验、官方文档查询、方案对比请求优先路由给 `researcher`。
3. 保持边界清晰：`researcher` 负责“先查信息”，不替代 `frontend` / `backend` / `plan` / `tech-lead` / `writer` / `qa_engineer` 的主职责。
4. 以最小规则集落地，避免引入复杂评分系统或过度建设。
5. 通过红灯测试先定义期望行为，再实现 production code。

## 非目标

1. 本次不同时补齐 `tech-lead` 独立语义路由。
2. 本次不将 analyzer 重写为 LLM 驱动的自由文本推理系统。
3. 本次不引入复杂的多维权重打分、置信度阈值或 capability matrix。
4. 本次不修改 Agent Definition Registry 的目录优先级、字段来源或 runtime dispatch 主链路。
5. 本次不重新设计全域 agent 路由框架，只做 `researcher` 的最小增量补齐。

## 用户确认的策略

用户已确认：

- 优先补 `researcher`
- 触发策略为 **中等触发**
- 设计原则为：既要减少漏判，也不能过度激进导致误把实现/规划类任务发给 `researcher`

这意味着本次不能只做“显式关键词直匹配”，也不能把所有需要思考或需要知识的任务都判给 `researcher`。

## 设计原则

1. **信息搜集优先，而非泛化思考优先**：只有当任务的主要目标是搜集、比对、核验外部信息时，才优先进入 `researcher`。
2. **中等触发，不保守也不激进**：既识别明确关键词，也识别研究型任务表达。
3. **强信号优先于弱信号**：当任务同时包含研究词与明确实现/文档/测试动作时，应优先尊重主目标角色。
4. **规则透明可测试**：采用显式规则与样例测试，保证可解释性。
5. **避免过度建设**：先做最小可用规则集，不引入评分系统、训练数据或复杂 DSL。

## 方案比较

### 方案 A：纯关键词直匹配

仅匹配：

- 调研
- 搜索
- 查资料
- 查文档
- 官方文档
- 对比方案

优点：

- 简单
- 可预测
- 测试编写成本低

缺点：

- 漏判较多
- 难覆盖“帮我看看业内怎么做”“先查一下官方推荐”这类研究型自然表达

### 方案 B：关键词 + 研究型意图双判断

同时识别：

- 明确研究词
- 研究型任务句式 / 意图短语

优点：

- 最贴合用户选定的“中等触发”
- 覆盖率与稳定性平衡较好
- 仍然可用显式测试约束，不需要复杂推理系统

缺点：

- 规则比纯关键词略多
- 需要补一批正反例测试，保证不越界

### 方案 C：宽松语义推断

只要看起来涉及外部事实、行业做法、方案比较，就尽量判为 `researcher`。

优点：

- 覆盖面最大
- 几乎不漏典型 research 请求

缺点：

- 很容易误伤 `plan`、`backend`、`tech-lead` 等角色
- 与当前“避免过度建设、保持角色边界”的方向不一致

## 方案选择

本次采用 **方案 B：关键词 + 研究型意图双判断**。

原因：

1. 与用户确认的“中等触发”完全一致。
2. 能较好覆盖自然语言中的 research 任务表达。
3. 仍然能保持 analyzer 的规则式、可测试、可解释实现。
4. 不需要引入新的复杂架构。

## 总体设计

### 一、为 analyzer 增加 `researcher` 域判定

在 `inferDomain(...)` 中引入一组 `researcherMatched` 规则，用于识别资料搜集、信息核验、官方资料查询、方案对比等任务。

识别信号分为两层：

#### 1. 明确研究关键词

建议覆盖以下最小词集：

- 调研
- 搜索
- 查资料
- 查文档
- 官方文档
- 官方推荐
- 对比方案
- 对比一下
- 搜集资料
- 收集资料
- 查一下
- 看看资料
- 业内怎么做
- 有哪些方案

#### 2. 研究型任务意图短语

建议覆盖以下表达方向：

- 帮我看看 X 怎么做
- 帮我看看业内一般怎么做
- 查一下官方怎么推荐
- 对比一下这几种方案
- 帮我搜集相关信息
- 先调研一下再给建议
- 有没有现成资料 / 案例 / 文档

这里的重点不是精确自然语言理解，而是识别“**请求的主目标是否为信息搜集与事实比对**”。

### 二、设置边界保护，避免抢占其他角色

即使命中部分 research 词，也不应无条件路由到 `researcher`。如果任务主目标已经非常明确，则仍然优先交给对应角色。

需要保留的主目标边界如下：

#### 仍应优先走实现角色

- “帮我实现一个鉴权中间件”
- “把这个插件接到 workflow 里”
- “写一个 API 来完成这个逻辑”

这类应继续走 `backend` / `frontend`，因为主目标是实现，不是研究。

#### 仍应优先走规划角色

- “给这个需求拆个实现计划”
- “帮我出方案”
- “分阶段规划一下怎么做”

这类应继续走 `plan` 或当前的 orchestration/pm 路径，因为主目标是规划，不是资料搜集。

#### 仍应优先走评审角色

- “帮我审一下这个架构方案”
- “做一个技术评审”

这类后续应走 `tech-lead`，不应被 `researcher` 抢占。

#### 仍应优先走文档角色

- “把这段说明整理成文档”
- “帮我更新 README”

这类应继续走 `writer`。

#### 仍应优先走测试角色

- “给这个模块补测试”
- “帮我回归验证一下”

这类应继续走 `qa_engineer`。

### 三、规则顺序建议

为保持行为稳定，推荐在 `inferDomain(...)` 中采用以下大致顺序：

1. `preferredAgent` 显式覆盖
2. orchestration 复合型强信号
3. 实现/文档/测试等已明确主目标的强信号
4. `researcher` 研究型信号
5. 默认回落到 orchestration

这里的关键不是“让 researcher 尽量早命中”，而是“**在不覆盖明确主目标的前提下，让 researcher 成为独立可命中的域**”。

换句话说，`researcher` 应位于“泛化默认回落之前”，但不应无条件压过更明确的实现、文档、测试等强信号。

### 四、推荐的最小行为样例

#### 应命中 `researcher`

- 帮我调研一下 React Native 埋点方案
- 查一下官方文档怎么推荐做鉴权
- 对比一下几种状态管理方案
- 帮我看看业内一般怎么做多租户权限
- 搜集一下这类插件的实现资料

#### 不应命中 `researcher`

- 帮我实现一个鉴权中间件
- 给这个需求拆一个实现计划
- 帮我审一下这个架构方案
- 把这段说明整理成文档
- 给我补一下这个模块的测试

## 对现有系统的影响

### 对 analyzer 的影响

- 需要在 `TaskDomain` / agent 选择链路上支持 `researcher` 作为独立语义域
- 需要更新 `mapDomainToAgent(...)` 映射
- 需要检查 `expectedNextAgents` 等派生逻辑是否允许 `researcher` 作为推荐角色

### 对 registry / runtime 的影响

- 无需修改 registry 规则
- 无需修改 runtime dispatch 主链路
- 只要 analyzer 输出 `researcher`，下游已具备基于 registry 解析实际 agent 定义的能力

### 对文档的影响

- 需要补充少量说明，明确 `researcher` 是“资料搜索/调研”角色，而不是实现角色
- 若存在 agent 指派表或 routing 说明，应同步更新该行为

## 风险与控制

### 风险 1：误判为 `researcher`

表现：一些原本要实现或规划的请求，因为包含“查一下”“对比一下”之类词而被误送 `researcher`。

控制：

- 保留主目标强信号优先级
- 用反例测试约束边界

### 风险 2：漏判自然语言 research 请求

表现：没有明确“调研”二字，但其实是在请求资料搜集。

控制：

- 补研究型任务意图短语
- 先覆盖高频表达，不追求一步到位全覆盖

### 风险 3：为后续 `tech-lead` 路由制造冲突

表现：技术评审类请求可能被 research 规则吞掉。

控制：

- 在 spec 中明确评审类请求不属于 `researcher`
- 后续实现 `tech-lead` 时保持强信号优先级更高

## 实施边界

本次实施只应包含：

1. analyzer 规则增量修改
2. 对应红灯测试与回归测试
3. 必要的最小文档同步

本次实施不应包含：

1. 大规模重写 analyzer
2. 引入复杂评分系统
3. 同时实现 `tech-lead` 的完整语义层
4. 改动 runtime / registry 的主设计

## 验收标准

1. analyzer 能独立识别典型 `researcher` 请求。
2. 中等触发策略成立：不只靠显式关键词，也能命中典型研究型意图短语。
3. 明确实现 / 文档 / 测试 / 规划主目标的请求不会被 `researcher` 抢占。
4. 所有新增测试先红后绿，并通过全量相关验证。
5. 设计保持最小增量，不引入新的复杂路由框架。
